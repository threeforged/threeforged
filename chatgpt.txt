Yep — that is the problem.

This line is what breaks installs from npm:

"@threeforged/core": "workspace:*"

workspace:* is for local workspace development, not for a consumer installing your published package from the npm registry. npm’s workspaces docs describe workspaces as local packages managed from a single top-level project, and pnpm’s docs note that workspace: dependencies are rewritten on publish/pack when you publish through pnpm.

What to change

Change this:

"dependencies": {
  "@threeforged/core": "workspace:*",
  "commander": "^12.0.0"
}

to something like:

"dependencies": {
  "@threeforged/core": "^0.1.1",
  "commander": "^12.0.0"
}

Use the actual published version of @threeforged/core.

Important question

Is @threeforged/core already published to npm?

Because if it is not published, then your asset analyzer package cannot be installed by outside users anyway. The dependency must exist on npm under a real version that consumers can download.

Why this happened

If you’re in a monorepo, workspace:* is fine inside the repo during development. But when publishing, the final published manifest needs normal semver dependency ranges for consumers. pnpm explicitly says it rewrites workspace: specs during pnpm pack / pnpm publish.

That means one of these likely happened:

you published with a flow that did not rewrite workspace:*

or the package that reached npm still contained the raw workspace reference

What to do next
If you want the fastest fix

Update package.json to use a real version:

"@threeforged/core": "^0.1.1"

Bump the plugin version:

"version": "0.1.2"

Rebuild and republish.

Before publishing, verify the packed manifest

Run this in the package folder:

npm pack --dry-run

or, if you use pnpm:

pnpm pack

pnpm documents that workspace: dependencies are replaced during pack/publish.
